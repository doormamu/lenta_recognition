# Lenta Price Recognition

Проект для автоматического распознавания ценников на видео с камеры робота.

Система принимает сырое видео, находит ценники с помощью YOLO, вырезает найденные области, распознает текст, сопоставляет товар со справочником и формирует итоговый CSV-файл в заданном формате.

---

## Основная идея

Пайплайн обработки:

```text
Сырое видео
  ↓
YOLO-детекция ценников
  ↓
detection_report.csv с bbox-координатами
  ↓
Вырезание crop-областей ценников
  ↓
Предобработка изображений для OCR
  ↓
Tesseract OCR
  ↓
Парсинг текста и сопоставление со справочником products.csv
  ↓
Постпроцессинг и нормализация полей
  ↓
Итоговый CSV
````

Пользователь через API загружает только видео.
Модель детекции и справочник товаров лежат локально внутри проекта.

## Используемые технологии

### Python

Основной язык проекта. Используется для реализации всех этапов пайплайна: чтение видео, детекция, OCR, парсинг, постпроцессинг и API.

### OpenCV

Используется для работы с видео и изображениями:

* чтение кадров из видео;
* получение `frame_index` и `timestamp`;
* вырезание crop-областей по bbox;
* поворот и улучшение изображений;
* сохранение debug-изображений.

### YOLO / Ultralytics

Используется для детекции ценников на исходных кадрах видео.

YOLO выбрана потому, что ценники имеют разный внешний вид:

* разные размеры;
* разные цвета;
* разные пропорции;
* разное положение QR/штрихкода;
* близкое расположение друг к другу;
* разное качество изображения.

На выходе YOLO возвращает bbox-координаты ценников:

```text
x_min, y_min, x_max, y_max, score, frame_index
```

Детекция выполняется на raw-кадрах, чтобы координаты оставались в системе координат исходного видео.

### Tesseract OCR

Используется для распознавания текста на найденных ценниках.

На вход OCR подаются подготовленные crop-изображения:

```text
crop
crop_rot90_ccw
```

То есть для каждого найденного ценника используется обычный crop и вариант, повернутый на 90 градусов влево.

### products.csv

Справочник товаров.
Используется для восстановления корректного названия и barcode.

Ожидаемые поля:

```text
fullname — полное название товара
code     — barcode / внутренний код товара
```

Так как barcode с изображения пока распознается нестабильно, основной способ получить barcode — сопоставить распознанное название товара со справочником.

### FastAPI

Используется как веб-интерфейс/API для запуска полного пайплайна.

Основная точка входа:

```text
POST /process
```

Пользователь загружает только сырое видео.
В ответ API возвращает итоговый CSV.

## Логика работы пайплайна

### 1. Загрузка видео

Пользователь отправляет видео в endpoint:

```text
POST /process
```

Видео сохраняется во временную папку:

```text
data/runtime/
```

Эта папка не хранится в Git, потому что содержит временные и тяжелые файлы.

---

### 2. Детекция ценников

На выбранных кадрах запускается YOLO-модель:

```text
models/detector/price_tag_detector.pt
```

Результат сохраняется в промежуточный файл:

```text
detection_report.csv
```

В нем содержатся координаты найденных ценников:

```text
frame_index
timestamp_ms
candidate_index
score
x_min
y_min
x_max
y_max
width
height
```

---

### 3. Подготовка изображений для OCR

Для каждого найденного bbox выполняется:

1. чтение исходного кадра;
2. frame-level preprocessing один раз на кадр;
3. пересчет bbox под обработанный кадр;
4. вырезание crop;
5. формирование OCR-вариантов:

```text
crop
crop_rot90_ccw
```

Такой подход быстрее, чем обрабатывать каждый crop отдельно с нуля.

---

### 4. OCR и парсинг

OCR выполняется через Tesseract.

После распознавания `field_parser.py` пытается извлечь из текста:

```text
product_name
price_default
price_card
discount_amount
id_sku
print_datetime
code
```

Для извлечения используются регулярные выражения и эвристики:

* поиск цен;
* поиск скидки вида `-X%`;
* поиск даты печати;
* поиск служебных кодов;
* выбор наиболее похожих на название строк.

---

### 5. Сопоставление со справочником

Файл `products.csv` используется для исправления названия и получения barcode.

Логика:

```text
распознанное название
  ↓
поиск похожего fullname в products.csv
  ↓
подтягивание нормального fullname
  ↓
подтягивание code как barcode
```

Для поиска используются:

* токенизация названия;
* частичное совпадение слов;
* триграммное сравнение;
* `SequenceMatcher`.

---

### 6. Определение цвета ценника

Цвет ценника определяется по crop-изображению.

Поддерживаемые значения:

```text
белый
желтый
красный
смешанный
-
```

Цвет определяется по HSV-маскам и доле цветных пикселей, особенно в нижней части ценника, где часто находится цветной промо-блок.

---

### 7. Постпроцессинг

Постпроцессинг приводит данные к единому формату:

* нормализует цены;
* нормализует скидки;
* нормализует barcode;
* нормализует координаты;
* очищает пустые значения;
* проверяет пару `price_default / price_card`;
* заполняет нераспознанные поля значением `-`.

Важно:

```text
barcode          — может быть получен из products.csv
qr_code_barcode  — заполняется только если QR/штрихкод реально считался с изображения
```

---

### 8. Экспорт CSV

Финальный файл формируется с фиксированным набором колонок:

```text
filename
product_name
price_default
price_card
price_discount
barcode
discount_amount
id_sku
print_datetime
code
additional_info
color
special_symbols
frame_timestamp
x_min
y_min
x_max
y_max
qr_code_barcode
price1_qr
price2_qr
price3_qr
price4_qr
wholesale_level_1_count
wholesale_level_1_price
wholesale_level_2_count
wholesale_level_2_price
action_price_qr
action_code_qr
```

Если поле не удалось распознать, записывается:

```text
-
```

## Ограничения текущей версии

1. OCR работает нестабильно на мелких и размытых ценниках.
2. Barcode с изображения пока почти не распознается.
3. Barcode в итоговом CSV в основном подтягивается из `products.csv`.
4. Качество результата сильно зависит от качества YOLO-детекции.
5. При большом количестве bbox распознавание работает медленно, потому что Tesseract запускается для каждого crop.
6. Справочник `products.csv` должен быть доступен локально.
7. Модель YOLO должна быть заранее обучена и помещена в `models/detector/`.

## Идеи для улучшения и оптимизации

### 1. Ускорение OCR

Самый медленный этап — Tesseract OCR.

Возможные оптимизации:

* уменьшить число кандидатов после YOLO;
* поднять `RECOGNITION_MIN_SCORE`;
* обрабатывать только top-N bbox на кадр;
* использовать только один вариант crop вместо двух;
* отключить чтение QR/штрихкодов, если оно не используется;
* распараллелить OCR по crop-областям;
* кэшировать обработанные кадры;
* не сохранять debug-crops в обычном режиме.

### 2. Улучшение детекции

Для повышения качества YOLO можно:

* добавить больше размеченных примеров;
* отдельно разметить мелкие ценники;
* добавить примеры красных, желтых и белых ценников;
* добавить сложные случаи: блики, наклон, соседние ценники, плохое качество видео;
* обучить более крупную модель YOLO;
* подобрать `imgsz`, `conf`, `iou`;
* использовать трекинг между кадрами, чтобы не распознавать один и тот же ценник много раз.

### 3. Улучшение распознавания текста

Возможные направления:

* настроить Tesseract на `rus+eng`;
* отдельно подбирать режимы `psm`;
* улучшить предобработку crop: бинаризация, контраст, deskew;
* использовать PaddleOCR или EasyOCR;
* обучить OCR под шрифт ценников;
* выделять только область названия товара, а не весь ценник.

### 4. Улучшение сопоставления со справочником

Сейчас используется fuzzy matching по названию.

Можно улучшить:

* использовать BM25 по `fullname`;
* использовать векторный поиск по названиям товаров;
* добавить нормализацию брендов;
* добавить словарь типичных OCR-ошибок;
* учитывать категорию товара;
* учитывать найденные цены;
* хранить несколько возможных совпадений со score, а не только одно.

### 5. Улучшение barcode / QR

Сейчас barcode с изображения почти не считывается.

Возможные улучшения:

* отдельно детектировать область QR/штрихкода;
* увеличивать именно область кода;
* использовать несколько библиотек декодирования;
* применять adaptive threshold;
* пробовать повороты и перспективное выпрямление только для зоны кода;
* обучить отдельную модель для поиска QR/штрихкода.

### 6. Оптимизация API

Для промышленного режима можно добавить:

* асинхронную очередь задач;
* сохранение статуса обработки;
* endpoint для получения прогресса;
* ограничение размера видео;
* очистку `data/runtime`;
* Docker volume для моделей и справочников;
* логирование каждого этапа;
* возврат не только CSV, но и JSON-отчета.

### 7. Улучшение финального CSV

Можно добавить:

* confidence по каждому полю;
* источник значения: OCR, справочник, QR, эвристика;
* несколько кандидатов товара из справочника;
* признак дубля ценника;
* номер кадра, где качество распознавания было максимальным;
* ссылку на debug-crop.
